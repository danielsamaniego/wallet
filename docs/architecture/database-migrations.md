# Database Migrations â€” Prisma

Approach for managing PostgreSQL schema and migrations in the Wallet Service using **Prisma Migrate**.

---

## Goals

- Auditable, ordered history of database changes
- Single schema source of truth (`prisma/schema.prisma`)
- Reproducible execution (local + CI/CD)
- No auto-migrate at runtime in production

---

## Tool

- **Prisma Migrate** (v7.x): `prisma migrate dev`, `prisma db push`, `prisma generate`
- **Schema**: `prisma/schema.prisma` â€” single source of truth for models
- **Config**: `prisma/prisma.config.ts` â€” connection URL for migrations (Prisma 7.x requirement)
- **Runtime**: `PrismaClient` uses the **Prisma Postgres adapter** (`@prisma/adapter-pg`) in `src/wiring.ts`. The adapter receives the connection string from config and is passed to `new PrismaClient({ adapter })`
- **Migrations**: Generated SQL in `prisma/migrations/` per migration

---

## Workflow

### Create migration (development)

```bash
pnpm db:migrate
# or: prisma migrate dev --name <migration_name>
```

Prisma generates:
- SQL migration files in `prisma/migrations/<timestamp>_<name>/`
- Applied migrations recorded in `_prisma_migrations` table

### Apply migrations (production / CI)

```bash
prisma migrate deploy
```

### Schema-only sync (dev prototyping)

```bash
pnpm db:push
# or: prisma db push
```

Use for rapid prototyping; does **not** create migration history. Prefer `migrate dev` for auditable changes.

### Generate client

```bash
pnpm db:generate
# or: prisma generate
```

Regenerates Prisma Client after schema changes.

## Audit Columns: created_at and updated_at

**All tables must include** `created_at` and `updated_at` as `BigInt` (Unix milliseconds). The app provides values on insert and update.

```prisma
createdAt BigInt @map("created_at")
updatedAt BigInt @map("updated_at")
```

---

## ID Generation â€” UUID v7 from Application

Entity IDs are **UUID v7** (RFC 9562). The **application** generates all IDs via `IDGenerator` (uuidv7) and provides them on every `create`. The database does **not** generate IDs.

- Do **not** use `@default(cuid())`, `@default(uuid())`, or `@default(dbgenerated())` for any primary key.
- **All** entity IDs (including `IdempotencyRecord`) use UUID v7, generated by the app via `IDGenerator`.
- App supplies the ID in every `create` call. No exceptions.

## Custom SQL: Immutable Ledger

After Prisma migrations, apply custom SQL for ledger immutability and safety constraints.

**Local development:**

```bash
pnpm db:constraints
# or as part of: pnpm db:update
```

**Production / CI:**

```bash
psql $DATABASE_URL -f prisma/immutable_ledger.sql
```

**Contents** (`prisma/immutable_ledger.sql`):
- **Trigger** on `ledger_entries`: Prevents UPDATE and DELETE; table is append-only.
- **Constraints**: `wallets` (positive balance unless `is_system`), `holds` (positive amount), `transactions` (positive amount).

This file is **not** part of Prisma migrations; it is applied manually or via a separate deploy step after migrations run. The SQL uses `DROP ... IF EXISTS` / `ADD CONSTRAINT` so it is idempotent and safe to re-run.

## Migration Naming

Prisma uses timestamp-based migration folders:

```
prisma/migrations/
  20240115120000_init/
    migration.sql
  20240116100000_add_hold_expiry/
    migration.sql
```

Use descriptive names: `add_hold_expiry`, `add_ledger_index`, etc.

## Production Practices

Production runs as a **plain Node.js process** against a managed PostgreSQL (Neon, Supabase, AWS RDS, etc.). No Docker.

### 1) Never auto-migrate in production

- Use `prisma migrate deploy --config prisma/prisma.config.ts` as a separate deploy step before app deployment.
- **Never** run `db:push` in production.
- **Never** run seed in production.

### 2) Deploy sequence

```bash
pnpm install --frozen-lockfile
pnpm db:generate
prisma migrate deploy --config prisma/prisma.config.ts
psql $DATABASE_URL -f prisma/immutable_ledger.sql
pnpm build
pnpm start
```

### 3) Large changes in 2 phases (expand/contract)

**Phase A (Expand):**
- Add nullable column or new table.
- Add indexes if needed.
- Deploy app that reads/writes new schema without breaking old.

**Phase B (Contract):**
- Backfill data (job/script outside migration if heavy).
- Harden constraints: set NOT NULL, add CHECK, etc.
- Remove legacy columns when safe.

### 4) Avoid long locks

- For large tables, consider `CREATE INDEX CONCURRENTLY` (requires custom SQL migration; cannot run inside Prisma transaction).
- Schedule heavy migrations outside peak hours.

### 5) Dirty state

If a migration fails, Prisma marks the database as dirty. Do not apply further migrations until the issue is fixed. Fix the migration or create a forward-fix before proceeding.

### 6) Startup verification

The app verifies on startup that the immutable ledger trigger and CHECK constraints exist. If any are missing, it logs a `FATAL` error and refuses to start. This is intentional â€” never disable this check.

## CI/CD Integration

Typical pipeline:

1. `pnpm install --frozen-lockfile`
2. `pnpm lint && pnpm test`
3. `pnpm build`
4. `prisma migrate deploy --config prisma/prisma.config.ts`
5. `psql $DATABASE_URL -f prisma/immutable_ledger.sql`
6. Deploy artifact / restart process

Avoid auto-running migrations on app startup in production; prefer explicit control.
